1. Zusammenfassung (Management Summary)

Der Artmodul Konfigurator ist eine webbasierte Anwendung, mit der Endkunden und Händler Regalsysteme der Marke Artmodul konfigurieren, eine Stückliste (BOM) berechnen und Anfragen an MHZ senden können.

Kernaussagen:

  • Kein Tracking, keine Analyse-Tools — die Anwendung setzt weder Google Analytics noch sonstige Tracking-Dienste ein.
  • Nur technisch notwendige Cookies — keine Einwilligungspflicht für Cookies nach ePrivacy-Richtlinie.
  • Drei Sicherheitsschichten — HTTP Basic Auth (Beta-Schutz) + Supabase Auth (Rollensystem) + Admin-Cookie-Auth (HMAC-signiert).
  • Row-Level Security (RLS) auf allen Datenbanktabellen — ergänzt durch DB-Trigger als Defense-in-Depth.
  • Personenbezogene Daten werden ausschließlich bei Bestellanfragen erhoben (Name, E-Mail, optional Adresse/Telefon).
  • Hosting: Vercel (Frontend/API, CDN global) + Supabase (Datenbank, EU-Region verfügbar).
  • Datenschutzerklärung und Impressum sind implementiert und aktuell (Stand April 2026).

3. Hosting & Infrastruktur

Vercel (Frontend + API)

  • Dienst: Vercel Inc., San Francisco, USA
  • Funktion: Hosting der Next.js-App (Frontend + Serverless API-Routen)
  • CDN: Globales Edge Network (statische Assets)
  • Serverless Functions: Node.js-Runtime für API-Routen
  • Verschlüsselung: TLS/HTTPS erzwungen (HSTS Preload)
  • Environment Variables: Verschlüsselt in Vercel-Projekteinstellungen
  • Logs: Serverless Function Logs (Vercel-Retention-Policy)

Supabase (Datenbank + Auth)

  • Dienst: Supabase Inc. (Open Source, AWS-basiert)
  • Funktion: PostgreSQL-Datenbank, Auth, Row-Level Security
  • Region: Konfigurierbar (EU-Region empfohlen, z.B. Frankfurt)
  • Verschlüsselung at Rest: AES-256 (AWS RDS)
  • Verschlüsselung in Transit: TLS 1.2+
  • Backups: Tägliche automatische Backups (Supabase Pro)
  • RLS: Auf allen Tabellen aktiviert

GitHub (Quellcodeverwaltung)

  • Dienst: GitHub Inc. (Microsoft), USA
  • Funktion: Versionskontrolle (Git), CI/CD-Trigger für Vercel
  • Personenbezogene Daten: Keine — Repository enthält nur Quellcode, keine Nutzerdaten
  • Secrets: .env.local ist via .gitignore ausgeschlossen, niemals im Repository
  • Zugriff: Privates Repository, nur autorisierte Entwickler

5. Sicherheitsarchitektur

5.1 Authentifizierung (3 Schichten)

SchichtMechanismusSchütztDetails
1. Beta-Schutz HTTP Basic Auth Gesamte Anwendung Benutzername/Passwort via BETA_USER / BETA_PASSWORD. Fail-Safe: 503 wenn nicht konfiguriert.
2. Supabase Auth Session-basiert (JWT) Händler- und Admin-Bereiche Rollen: customer, dealer, admin. Session-Erneuerung bei jedem Request.
3. Admin-Cookie HMAC-SHA256 signiert /admin/* Routen 8h Gültigkeit. HttpOnly, Secure, SameSite=Strict. Timing-Safe Verifikation.

5.6 Input-Validierung

Alle API-Endpunkte validieren eingehende Daten mit Zod-Schemas. Ungültige Anfragen werden mit HTTP 400 und detaillierter Fehlermeldung abgelehnt. Zusätzlich:

  • Konfigurations-Codes: Regex-Prüfung (/^\d{8}$/)
  • Alle Benutzereingaben werden getrimmt (Whitespace-Bereinigung)
  • E-Mail-Templates nutzen Entity-Encoding (esc()) gegen XSS

8. Offene Punkte & Handlungsempfehlungen

8.1 DSGVO / Rechtlich

⚠
Auftragsverarbeitungsverträge (AVV) abschließen
Priorität: Hoch
Mit Vercel und Supabase müssen formale AVV gemäß Art. 28 DSGVO vorliegen. Beide Anbieter stellen Standard-DPAs bereit, die geprüft und akzeptiert werden müssen. Zuständig: DSGVO-Verantwortlicher.
⚠
Verzeichnis der Verarbeitungstätigkeiten (VVT)
Priorität: Hoch
Ein formales VVT gemäß Art. 30 DSGVO muss erstellt werden. Dieses Briefingpapier liefert die inhaltliche Grundlage.
⚠
Supabase-Region auf EU festlegen
Priorität: Hoch
Es sollte sichergestellt werden, dass das Supabase-Projekt in einer EU-Region (z.B. eu-central-1 Frankfurt) gehostet wird. Falls aktuell US-Region: Migration planen.
ℹ
Google CDN für model-viewer (AR-Seite) in Datenschutzerklärung ergänzen
Priorität: Mittel
Die AR-Vorschau-Seite lädt ein Script von ajax.googleapis.com. Die IP-Adresse des Nutzers wird dabei an Google übermittelt. Dies sollte in der Datenschutzerklärung erwähnt werden. Alternativ: Script lokal hosten.
ℹ
Self-Service für Betroffenenrechte (optional)
Priorität: Niedrig
Aktuell werden Auskunfts-, Lösch- und Exportanfragen manuell bearbeitet. Bei steigendem Aufkommen könnte ein Self-Service-Endpunkt sinnvoll werden.

8.2 Technisch / Sicherheit

⚠
Rate Limiting auf Redis umstellen (Produktion)
Priorität: Mittel
Aktuell In-Memory-Store — wird bei Serverless-Neustart zurückgesetzt und ist nicht instanz-übergreifend. Empfehlung: Upstash Redis (Vercel-kompatibel) für persistentes Rate Limiting.
ℹ
CSP: unsafe-inline / unsafe-eval perspektivisch ersetzen
Priorität: Niedrig
Für React/Next.js aktuell erforderlich. Mittelfristig Nonce-basierte CSP evaluieren, um XSS-Schutz zu verbessern. Kein akuter Handlungsbedarf.
ℹ
Beta-Schutz: Timing-Safe-Vergleich
Priorität: Niedrig
HTTP Basic Auth verwendet direkten String-Vergleich (kein timingSafeEqual). Theoretisches Timing-Angriffs-Risiko, praktisch irrelevant für Beta-Passwort.
ℹ
Server-Logging / Monitoring
Priorität: Niedrig
Aktuell nur Vercel-Standard-Logs. Für Produktion ggf. strukturiertes Logging (z.B. fehlgeschlagene Admin-Logins) einrichten.

8.3 Funktional / Business

⚠
Beta-Modus vor Go-Live entfernen oder anpassen
Priorität: Vor Go-Live
HTTP Basic Auth ist für die Betaphase vorgesehen. Vor Go-Live muss entschieden werden: entfernen (öffentlich) oder durch andere Zugangskontrolle ersetzen.
ℹ
E-Mail-Zustellung sicherstellen
Priorität: Mittel
SMTP ist optional konfiguriert. Vor Go-Live sicherstellen, dass Bestellbestätigungen und Admin-Benachrichtigungen zuverlässig zugestellt werden. Ggf. Retry-Mechanismus oder Queue einrichten.
